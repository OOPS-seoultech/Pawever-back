# 암호화 키 로테이션 절차 (참고)

Pawever는 개인정보(PII)를 애플리케이션 레벨에서 암호화·해싱한다.
**키 버전 태깅을 두지 않는다(설계상 선택)** — 복호화는 잘못된 키에 대해 즉시 실패(fail-fast)하며, 문제를 조용히 숨기지 않는다. 대신 키를 교체하려면 **기존 데이터 재처리 마이그레이션**이 필요하다. 이 문서는 그 절차를 정리한다.

## 키 종류

| 환경변수 | 용도 | 로테이션 영향 |
|---|---|---|
| `ENCRYPTION_SECRET_KEY` | User `name`/`phone`/`email`, 리뷰 `content` 등 **AES-256-GCM 암호화** | 교체 시 기존 암호문 **전량 재암호화** 필요 |
| `ENCRYPTION_HASH_KEY` | phone/email **블라인드 인덱스(HMAC-SHA256)** | 교체 시 기존 해시 **전량 재계산** 필요 |
| `JWT_SECRET` | 액세스 토큰 서명(HS256) | 교체 시 기존 토큰 **전부 무효화**(재로그인) — 마이그레이션 불필요 |

> 운영 기동 시 `ProdSecretsValidator`가 위 값들이 공개 기본값이거나 비어 있지 않은지 검증한다(fail-fast).

## 왜 즉시 교체가 안 되나

- `AesEncryptor.decrypt`는 잘못된 키/손상된 값에 대해 **예외를 던진다**(빠른 실패). 키만 바꾸면 기존 행 조회가 전부 실패한다.
- 암호문에 키 버전이 없어 이전/현재 키를 동시에 시도할 수 없다.

## 절차 — `ENCRYPTION_SECRET_KEY` / `ENCRYPTION_HASH_KEY`

1. **DB 백업** (`scripts/backup-db-to-object-storage.sh` 또는 덤프).
2. 트래픽이 적은 시간대 선택 또는 쓰기 일시 중단(점검 모드).
3. **재처리 러너 실행** — 구(old) 키로 복호화 → (해시는 정규화) → 신(new) 키로 재암호화·재해시 → 저장.
   - 대상 컬럼: `users`(`name`, `phone`, `phone_hash`, `email`, `email_hash`, `apple_refresh_token`), `reviews`(`content`) 등 암호화/해시 컬럼.
   - 러너는 old/new 키를 **모두 주입**받아야 한다(예: 임시 `ENCRYPTION_SECRET_KEY_OLD` 환경변수 추가).
4. 신 키를 배포하고 앱 재기동.
5. 표본 계정으로 로그인·프로필 조회를 검증한 뒤 점검 모드 해제.

> 재처리 러너는 현재 코드에 포함돼 있지 않다(로테이션 계획 발생 시 작성). 한 번 쓰고 제거하는 **게이트된 `ApplicationRunner`**(프로퍼티/프로필로 켜고 끄는 형태)를 권장한다.

## 절차 — `JWT_SECRET`

마이그레이션 불필요. 신 키 배포 후 기존 토큰은 서명 검증에 실패하므로, 클라이언트가 재로그인하면 된다. **강제 로그아웃**이 발생하므로 사용자 공지만 필요.
